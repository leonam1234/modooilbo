# 02 · 측정·성장·수익 (3순위)

> 매체를 "운영"하려면 트래픽을 **측정**하고, 독자를 **재방문**시키고(뉴스레터), **수익**을 내고(구독·광고·후원), **참여**시켜야(댓글) 한다. 측정은 즉시 가능, 수익·댓글 일부는 Workers 승급이 필요하다.

---

## ⑦ 측정 인프라 (애널리틱스 / GSC) — 즉시 가능, 최우선 착수

### 왜 필요한가
**데이터 없이는 개선 방향을 알 수 없다.** 어떤 기사가 읽히는지, 유입 채널, 검색 쿼리, CWV 실측이 없으면 모든 후속 작업이 깜깜이.

### 현재 상태
- `src/app/layout.tsx`에 Cloudflare beacon 흔적 존재(정식화 필요).
- google-site-verification 토큰 존재(GSC 연결 가능 상태).

### 정적 export 제약
없음. 클라이언트 스크립트만 삽입.

### 구현 방향
1. **Cloudflare Web Analytics**(쿠키리스·개인정보 친화) 또는 **GA4**. 한국 운영이면 GA4 + (선택) 네이버 애널리틱스.
2. **삽입**: `next/script`(strategy="afterInteractive")로 layout에. 동의(쿠키배너)는 PIPA 고려.
3. **GSC/네이버 서치어드바이저**: sitemap·news-sitemap 제출(→ [00-B](00-prerequisites.md)), Core Web Vitals·색인 커버리지 모니터.
4. **이벤트**: 기사 스크롤뎁스·구독전환·공유 클릭 등 커스텀 이벤트.

### 완료 기준 ✅
- [ ] 애널리틱스 라이브(실측 페이지뷰)
- [ ] GSC·네이버 서치어드바이저 연결, sitemap 제출·색인 추적
- [ ] CWV 대시보드(모바일 LCP/CLS/INP)
- [ ] 쿠키/개인정보 동의 처리(PIPA)

### 위험
- 동의 없는 추적·개인정보 수집은 PIPA 위반 — 쿠키리스 우선 또는 동의배너.

## ⑧ 뉴스레터 실연동

### 왜 필요한가
검색·소셜 유입은 휘발성. **뉴스레터 = 직접 소유 채널**(재방문·구독 전환·매체 충성). 00-direction의 핵심 수익 경로.

### 현재 상태
- `src/components/NewsletterCTA.tsx`,`src/app/newsletter/NewsletterToggle.tsx` — **클라이언트 데모(백엔드 전송 없음)**.

### 정적 export 제약
서버 액션/route handler 불가 → **외부 ESP 폼 액션** 또는 ESP API 직접 호출(클라이언트)·외부 임베드.

### 구현 방향
1. **ESP 선택**: Stibee(국내·한글친화) / ConvertKit / Resend / Mailchimp.
2. **연동**: 폼 `action`을 ESP 엔드포인트로(POST) 또는 클라이언트 fetch. 더블옵트인(스팸방지·PIPA).
3. **발송 콘텐츠**: "모두의 아침" 등 — RSS-to-email 자동화 또는 큐레이션.
4. **세그먼트**: 카테고리별 구독.

### 완료 기준 ✅
- [ ] 구독 폼이 실제 ESP에 저장(더블옵트인)
- [ ] 정기 발송 운영(자동/수동)
- [ ] 구독 동의·수신거부(법적)

### 위험
- 데모 폼을 운영에 그대로 두면 구독자 유실·신뢰 훼손 — 연동 전 "준비중" 명시.

## ⑨ 수익화 (구독결제·광고·후원)

### 왜 필요한가
지속가능성. 00-direction은 **후원·구독 우선, 콘텐츠 개방** 지향. `/subscribe`(요금제 3종) 골격 존재.

### 현재 상태
- `/subscribe`,`/advertise` 페이지·폼 데모. 결제·정산 미연동.

### 정적 export 제약
- 결제 **콜백/웹훅**은 서버 필요 → 외부 PG 위젯/호스티드 결제 페이지로 리다이렉트하거나 Workers 승급.

### 구현 방향
1. **구독결제**: 토스페이먼츠/포트원(국내) 또는 Stripe. 호스티드 결제 → 성공 리다이렉트. 정기결제는 PG 빌링키.
2. **광고**: 초기 Google AdSense(자동) → 트래픽 성장 후 직접판매(디스플레이/네이티브/브랜디드, `/advertise` 실연동).
3. **후원**: 일회/정기 후원(PG) 또는 외부(Buy Me a Coffee류).
4. **회원/페이월**: 인증 필요 → Workers 승급([06 §5](../06-deployment.md)). 초기엔 소프트 페이월(미터링) 지양, 후원 우선.

### 완료 기준 ✅
- [ ] 구독/후원 결제 실제 완료·영수
- [ ] 광고 게재(정책·레이아웃 CWV 영향 점검)
- [ ] 결제·환불·정산 운영 프로세스

### 위험
- 광고 스크립트가 LCP/CLS를 망가뜨려 SEO 역행 — 지연로딩·예약공간 필수. 결제는 PCI/보안 위해 직접 카드처리 금지(PG 위임).

## ⑩ 댓글·커뮤니티

### 왜 필요한가
참여·체류·재방문. 단 운영부담(악플·법적책임)이 큰 영역.

### 현재 상태
- 미구현. ethics에 "댓글·커뮤니티 정책" 항목 텍스트 존재.

### 정적 export 제약
- 자체 댓글(작성/저장)은 동적 → Workers 승급 필요. **외부 위젯은 정적에서 즉시 가능**.

### 구현 방향
1. **초기(정적)**: 외부 위젯 — Disqus / Giscus(GitHub) / Cusdis(개인정보 친화). 기사 하단 클라이언트 임베드.
2. **고도화(Workers)**: 자체 댓글 + 인증 + 신고/모더레이션.
3. **정책**: 실명/약관·신고·삭제 기준 ethics 반영. 청소년보호.

### 완료 기준 ✅
- [ ] 기사 하단 댓글(초기 외부 위젯)
- [ ] 신고·모더레이션·금칙어
- [ ] 댓글 정책 ethics 명문화

### 위험
- 모더레이션 없는 댓글은 명예훼손·법적책임·브랜드 훼손 — 운영인력 없으면 도입 보류가 안전.
